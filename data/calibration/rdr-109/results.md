## Sweep results

Hit rate = fraction of QA items where the expected chunk's chash appears in top-5 results after applying boost.

| content_type | baseline (w=0) | w=0.025 | w=0.05 | w=0.075 | w=0.10 | w=0.15 | Pareto-clean winner |
|---|---:|---:|---:|---:|---:|---:|---|
| knowledge | 0.286 | 0.343 (-2 reg) | 0.343 (-2 reg) | 0.343 (-2 reg) | 0.343 (-2 reg) | 0.343 (-2 reg) | **NONE** |
| rdr | 0.343 | 0.343 | 0.343 | 0.343 | 0.343 | 0.343 | none needed (no movement) |
| code | 0.486 | **0.514** | 0.514 | 0.514 | 0.514 | 0.514 | **w=0.025** |
| docs | 0.429 | **0.486** | 0.486 | 0.486 | 0.486 | 0.486 | **w=0.025** |

The "-2 reg" annotation indicates two QA items that hit at baseline w=0 miss at the boosted weight (Pareto regression).

## Interpretation

- **Boost plateaus immediately above zero.** All weights ≥ 0.025 give identical hit rates because the token-overlap fraction is tiny (typically 1-3 overlapping tokens out of ~10 query tokens), so any non-zero weight applies a uniform small nudge that determines rank ties regardless of magnitude.
- **Pareto-clean improvement on code + docs.** On both these corpora, w=0.025 wins by +1 hit (code) or +2 hits (docs) with zero baseline-regressions.
- **rdr is neutral.** No movement either direction. The 130-chunk corpus may be too small for the boost to reorder anything within top-5.
- **knowledge regresses 2 queries at every non-zero weight.** Aggregate is +2 hits net, but the Pareto rule (no baseline-passing query may miss after boost) rejects every weight.

## Caveat: programmatic Q&A

Questions were generated by `scripts/rdr-109-generate-qa.py` from real chunk content via templates ("What discusses: ...?", "How is X implemented?"). This produces verifiable expected chashes but yields near-paraphrase questions whose retrieval characteristics may differ from hand-curated benchmark questions. The aggregate trend is informative; absolute hit rates should be interpreted within this caveat.

## Decision

Per the bead's acceptance criterion ("pick boost weight that maximizes hit rate WITHOUT regressing any baseline-passing query") and per RDR-109 §coupling note (lines 290-293, split-if-blocked clause):

**Recommended Phase 5 default**: **w=0.025**, gated behind `attention_guided_v1.enabled` feature flag with **default OFF**. The bead's mandate that Phase 5 default-on only if measurements pass is **not met** for knowledge corpora — the boost regresses 2 baseline-passing queries at every weight tested.

Rationale:

- Pareto-clean wins on code + docs justify shipping the mechanism.
- knowledge regression is a real signal, not an artifact of programmatic Q&A — the trade-off (4 new hits, 2 lost) suggests the boost reorders within a narrow window that hand-curated QA would also be affected by.
- Opt-in via feature flag lets motivated operators evaluate per-corpus.
- Hand-curated QA expansion (still tracked in the original Phase 4b bead scope) could revisit the default-on decision.

## Reproducibility

```bash
# Regenerate QA (deterministic with seed=109):
python scripts/rdr-109-generate-qa.py \
    --content-type knowledge --collection knowledge__rag-papers --count 35

# Run sweep:
python scripts/rdr-109-calibrate.py \
    --content-type knowledge \
    --collection knowledge__rag-papers \
    --top-k 5 \
    --out data/calibration/rdr-109/results_knowledge.json
```

Per-content-type JSON results are committed at `data/calibration/rdr-109/results_<ct>.json`.
