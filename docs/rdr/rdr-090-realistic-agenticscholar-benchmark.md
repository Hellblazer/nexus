# RDR-090: Realistic AgenticScholar Benchmark

The AgenticScholar paper reports NDCG@3=0.606 vs. naive RAG 0.411 on a
25-document synthetic corpus with LLM-generated queries and ground
truth. RDR-042's critique flagged this as self-validating: the same
system that designed the index designed the queries. Nexus now runs
against real corpora (ART ~15,790 docs; Delos ~1397; RDR corpora
across 7 projects) and has no empirical discriminator between its
plan-first pipeline and a naive search baseline. This RDR proposes
a human-authored benchmark on real Nexus corpora with a three-way
comparison designed to isolate plan-first value.

## Problem Statement

### Enumerated gaps to close

#### Gap 1: No empirical discriminator between plan-first and naive RAG on real corpora

`nx_answer` (plan-first) and `search` (naive) both exist. No A/B
measurement shows which wins, on which query class, under what
threshold settings. Capability claims ("we handle multi-hop") are
unsubstantiated.

#### Gap 2: No regression guard on retrieval quality

RDR-088 introduces three new operators; RDR-089 adds an ingest-time
enrichment stage; future plan library promotions change match
behavior. Without a baseline bench, all of these risk silently
degrading retrieval quality, with the first signal being user
complaints.

#### Gap 3: Paper's benchmark is not reproducible or transferable

Its LLM-query-generation procedure depends on proprietary prompts
and a corpus not shared. Numbers cannot be compared with Nexus's;
we need Nexus-native numbers anyway. This gap is about building
our own, not reproducing theirs.

## Context

### Background

The AgenticScholar retrospective (2026-04-17) flagged three problems
with the paper's evaluation: (1) queries were LLM-generated from the
same corpus; (2) ground truth was LLM-labeled with template constraints
("must match 2-10 papers"); (3) baselines were proprietary systems
(Elicit, Gemini Deep Research) with their own training corpora,
making attribution impossible. The 47% relative improvement claim
is unfalsifiable from the paper alone.

Nexus's position: ART and RDR corpora are human-authored content
indexed over ~3 months of real work. Real queries against this
corpus already exist in session logs (`nx search` telemetry under
RDR-087). Human-authored ground truth is feasible at 30-query scale.

### Technical Environment

- `src/nexus/search_engine.py` — naive baseline (path A).
- `src/nexus/mcp/core.py` (`nx_answer`) — plan-first with plan_match (path B). (Line numbers in this section are intentionally omitted; they drift across releases.)
- `src/nexus/plans/runner.py` — target module for the `force_dynamic` skip-plan-match path (path C). The `force_dynamic` flag is a Phase 1 deliverable and does not yet exist; the spike used `scope="<full collection name>"` as a forced-miss proxy.
- `docs/rdr/rdr-087-collection-observability-surfaces.md` — telemetry
  infrastructure this bench can plug into.

## Research Findings

### Investigation

Paper §6.1–6.2: benchmark construction uses template-constrained LLM
query generation. §6.2 acknowledges circularity implicitly ("novel,
corpus-specific information" generated by a system trained on that
corpus).

Current state: no Nexus benchmark. `tests/` has unit + integration
tests for retrieval correctness but no quality measurement. RDR-087
retrieval telemetry will provide per-query distance data that
bench can cross-reference.

### Dependency Source Verification

| Dependency | Source Searched? | Key Findings |
| --- | --- | --- |
| `scikit-learn.metrics.ndcg_score` | Yes | Stable API, handles zero-match queries gracefully. Fit for NDCG@3. |
| `nx search --json` | Yes (`src/nexus/commands/search_cmd.py`) | Returns ranked chunk list with scores; usable as bench input. |
| `nx_answer` MCP tool | Yes (`src/nexus/mcp/core.py:1773`) | Returns structured answer + citations; needs parsing to extract ranked chunks. |

### Key Discoveries

- **Verified** (search_cmd.py): `nx search --json` is stable for
  programmatic consumption.
- **Verified** (search_engine.py): baseline path uses
  `voyage-context-3` embedding + threshold filter; deterministic
  given fixed query and corpus.
- **Assumed → partially informed by spike**: 30 human-authored queries
  are sufficient to discriminate path A/B/C with statistical
  confidence. The 5-query spike (research-1002) produced A2 FAIL
  (only 2 of 5 queries above the |ΔNDCG@3| ≥ 0.05 threshold; criterion
  required 3 of 5). N=30 (10 per category) provides sufficient
  expected variation across query types for directional signal;
  formal power analysis deferred pending actual per-query variance
  from the bench's first run.
- **Assumed**: multi-hop precision is measurable with small-N
  compositional queries. Paper reports on N=10 compositional
  queries; our 10 should be enough for directional signal, not
  formal significance.

### Critical Assumptions

- [x] Human-labeled ground truth is feasible at 30-query scale within a 2-day labeling effort — **Status**: VERIFIED (research-1001/1002) — 5 queries authored + GT in ~12 min from project memory; extrapolated 30 × 5–10 min ≈ 2.5–5 hours, well under the 16h budget. Q2 surfaced the chunk-aware-authoring requirement (zero NDCG@3 on all paths when GT vocab mismatched corpus chunk vocabulary) which inflates the per-query budget but stays inside it.
- [ ] Path A vs. path B vs. path C shows measurable NDCG@3 deltas on at least 50% of queries — **Status**: FAILED at N=5 (research-1002) — only 2 of 5 queries reached the pre-registered |ΔNDCG@3| ≥ 0.05 threshold; criterion required 3 of 5. Failure at N=5 is a definitive signal that the 5-query spike subset is too small to discriminate paths reliably; the proposed N=30 (10 per category) remains the right scale and is more credible after this finding, not less. Bench proceeds.
- [ ] Compositional queries can be answered by path C (forced-dynamic) without plan_match assistance, enabling the three-way split — **Status**: PREMISE WEAKLY SUPPORTED (research-1002) — Q5 returned non-zero NDCG@3=0.469 on all paths, but plan #38 (the matched plan) is a 2-step search → summarize that reduces path C to a thin wrapper around path A on this category. True path-C value cannot be measured until the `force_dynamic` flag lands and the plan library carries genuinely multi-step compositional plans (see Path C contingency note in *Decision Rationale*).

## Proposed Solution

### Approach

Build a lightweight benchmark harness in `scripts/bench/` that:
1. Loads a YAML-defined query set (30 queries, 3 categories of 10).
2. Runs each query through three paths (A: naive, B: plan-first, C: forced-dynamic).
3. Computes NDCG@3 against human-labeled ground truth plus multi-hop precision on compositional queries.
4. Outputs a JSON report with per-query and aggregate metrics.
5. CI-runnable with a quality-regression alert on ≥10% NDCG@3 drop.

### Technical Design

**Query categories** (10 each):
- **Factual**: single-hop, single-chunk answer. "Which RDR introduced catalog tumblers?"
- **Comparative**: two-chunk, different-document. "Compare the RDR-049 catalog model with the paper's KG approach."
- **Compositional**: three+-hop, requires plan composition. "What retrieval architecture decisions were made across RDR-078, 079, 080, and what was the sequence of tradeoffs?"

**Ground truth format** (`bench/queries/*.yaml`):

```text
# Illustrative — schema defined in bench harness
query: "Which RDR introduced catalog tumblers?"
category: factual
corpus: rdr__nexus-571b8edd
ground_truth:
  - chunk_id: rdr-049-*-chash:abc123...
    relevance: 3  # graded 0-3
  - chunk_id: rdr-049-*-chash:def456...
    relevance: 2
acceptance:
  ndcg_at_3: 0.8  # pass threshold
```

**Three paths**:
- **A (naive)**: `search(query, corpus, limit=3)` — top-3 chunks, no plan.
- **B (plan-first)**: `nx_answer(question=query)` — via plan_match, then plan_run.
- **C (forced-dynamic)**: `nx_answer(question=query, force_dynamic=True)` — skip plan_match, always generate plan. **New flag**; minor change to `nx_answer`.

**Metrics**:
- NDCG@3 per query, aggregated by path and category.
- Multi-hop precision (compositional only): fraction of queries where all required documents are retrieved.

**Regression alert**: CI job runs the bench weekly; alerts if NDCG@3
drops ≥10% on any category vs. last saved baseline.

| Proposed Component | Existing Module | Decision |
| --- | --- | --- |

### Decision Rationale

Three-way split (naive / plan-first / forced-dynamic) directly
discriminates plan-first value. The paper's single-system vs.
proprietary-baseline comparison confounds system design with
baseline provenance.

Human-authored queries (vs. LLM-generated) eliminates the circularity
the paper's benchmark has. 30 queries is the smallest N that gives
10 per category, enough for directional signal.

**Path C compositional contingency** (research-1002 finding): path C's
compositional discrimination is contingent on the plan library
containing genuinely multi-step compositional plans — not 2-step
search → summarize wrappers. Without those, path C collapses to path
A on the compositional category (Q5 spike: A=B=C=0.469 because the
matched plan #38 was such a wrapper). This is a known state the
bench must annotate per-query, not a design flaw in the three-way
split itself. Phase 1's `force_dynamic` flag lands the mechanism;
Phase 2's compositional queries will reveal whether the live plan
library has the depth required for path C to differentiate.

## Alternatives Considered

### Alternative 1: Reproduce the paper's benchmark on our corpus

**Description**: Use the paper's 25 synthetic survey-paper corpus +
LLM-generated queries + LLM-labeled ground truth.

**Pros**: Directly comparable to paper's numbers.

**Cons**: Inherits the circularity. Doesn't measure anything about
Nexus on real workloads.

**Reason for rejection**: The critique we levied at the paper applies
equally to any reproduction of its method.

### Alternative 2: Use agent-generated queries at larger scale

**Description**: LLM-generate 300 queries, LLM-label ground truth,
get statistical power.

**Pros**: Scales; cheap.

**Cons**: Same circularity the paper has.

**Reason for rejection**: Defeats the purpose.

### Briefly Rejected

- **Benchmark all MCP tools**: scope bloat; keep bench focused on retrieval quality.
- **Include latency as a ranking criterion**: separates concerns; bench measures quality, not speed.

## Trade-offs

### Consequences

- 30 queries + ground truth is ~2 days of human effort.
- Adds CI weight (~30 queries × 3 paths × avg 2s/query ≈ 3 minutes per bench run).
- Creates a quality baseline that any future change can be measured against.
- Forces `force_dynamic` flag on `nx_answer` — low-risk, opt-in.

### Risks and Mitigations

- **Risk**: Ground-truth labeling bias from the author (Hal is the primary labeler, also the author of most RDRs being queried).
  **Mitigation**: Cross-corpus queries where GT is less biased (ART + Delos, not RDR only); future third-party review of 20% sample.

- **Risk**: Bench becomes stale as corpora grow.
  **Mitigation**: Versioned corpus snapshots referenced in query YAML; bench re-validates applicability on each run.

- **Risk**: Compositional queries' GT is inherently subjective (which documents are "required"?).
  **Mitigation**: Coarser metric on compositional — "did the system retrieve at least 3 of 5 listed documents?" rather than exact ranking.

### Failure Modes

- **Visible**: path C (forced-dynamic) fails to return answers on compositional queries → measurable as coverage 0 on that category.
- **Silent**: threshold calibration drift causes path A to dominate spuriously → detected by cross-checking against the expected-deltas spike.

## Implementation Plan

### Prerequisites

- [x] Spike: author 5 queries with GT; measure labeling time and delta-between-paths on a 5-query subset. **Done — research-1001/1002/1003.**
- [ ] **Add `force_dynamic` flag to `nx_answer`** — small change to `src/nexus/mcp/core.py`. Without it, path C cannot be isolated from path B for collection-scoped questions.
- [ ] **Plan-library hygiene** (research-1002 side-findings) — file beads to (a) tighten `plan_match` so a plan with hardcoded `corpus="rdr__<other-project>-..."` does not match a `scope` for a different project's RDR collection (plan #52 leakage at confidence 0.476); (b) tighten the match-text generation in RDR-084 plan-grow so over-broad plans like #67 (returns "tumblers" for every "which RDR" question) cannot pollute the library. Without these, every bench run re-confirms path-B noise from the same two known pollutants.
- [ ] **T3 staleness sweep** — promote `scripts/spikes/spike_rdr090_remediate_corpus.py` to a `nx` subcommand or CI check that deletes T3 chunks whose `source_path` is missing from disk. Spike found 7.8% of `rdr__nexus-571b8edd` was stale; the long tail surfaces at N=30+ even though it didn't pollute N=5 top-3.
- [ ] **Catalog staleness remediation** — extend `nx catalog remediate-paths` (RDR-326) to a "match by RDR-number prefix" mode, OR ship `nx catalog prune-stale` for catalog rows whose `file_path` is missing from disk and has no plausible same-prefix replacement. 12 entries currently produce null-field aspect rows on every `nx enrich aspects rdr__nexus-...` run.
- [ ] **Chunk-aware query authoring discipline** — bench authoring section must mandate top-K inspection during query authoring (refine query, not GT, when expected RDR doesn't surface). Q2 lesson — universal NDCG@3=0 because GT vocab "content-hash chunk index" did not match indexed chunk vocab.

all three paths and produces a JSON report with NDCG@3 and raw rankings.
