# 6.0.0 Plugin Claude-Surface Coverage Matrix

Living map of behavioral test coverage for every Claude-facing conexus surface, built 2026-06-27 (parallel inventory). Companion to T2 `nexus/6.0.0-plugin-surface-coverage-gap` [4733] and `nexus/6.0.0-test-coverage-gap-map` [4732].

**Coverage legend** (two dimensions that matter for a plugin release):
- `fn/unit` — the tool/hook logic is exercised in-process (imports the fn / drives the HTTP or unit harness, asserts behavior). Real, but NOT a real Claude session and NOT necessarily against the 6.0.0 PG service.
- `real-session` — fired in a REAL `claude` session (cc-validation scenario, `tests/e2e/scenarios`, or `migration-rehearsal --fullstack`).
- `structure-only` — only `test_plugin_structure.py` static checks (frontmatter/sections/naming/registry).
- `none` — no test of any kind.

**Headline:** MCP tools are well-covered at the `fn/unit` path (43/45); commands (0), skills (1/45), agents (0/13) have ~no behavioral coverage; hooks are partially `fn/unit`-covered with named gaps. **Almost nothing is `real-session` against the 6.0.0 service** — that's what `--fullstack` + cc-val expansion must close.

## Strategy: MULTIVARIATE functional scenarios (NOT 1:1 per-surface)

Per-surface testing of ~150 surfaces is infeasible and unrepresentative. The discipline is **multivariate**: a small set of realistic functional Claude sessions, each traversing a WIDE SWATH of surfaces together, chosen so their **union** covers the surface (Tier-1 first, then all). One rich session exercises hooks + skills + commands + MCP tools + agents + the service in the way users actually do — and surfaces integration bugs that isolated tests never see. Coverage is measured as the union of surfaces touched + asserted across the scenario set.

### Scenario set (design for maximal union coverage)

**M1 — Dev loop: research → RDR → implement → review** (cc-val, real session)
Fires: SessionStart hooks (session_start, rdr_hook, using-nx-skills inject, preflight, version_lockstep) → using-nx-skills gate → query/research skills → search/query/nx_answer/traverse MCP tools → rdr-create/gate/accept commands+skills (`nx rdr preamble`) → memory_put → substantive-critic + developer + code-review-expert AGENTS → routing PreToolUse (git_add_all, grep_for_symbols→serena) → Stop verification hook.
Swath: ~half the skills, ~6 commands, 3-4 agents, hot tools, most hooks.

**M2 — Knowledge ingest + analytical QA + queue** (`--fullstack` + cc-val, real session vs 6.0.0 service)
Fires: store_put/index via MCP → post-store hook → aspect enqueue → MCP worker drain (real extraction) → query/analyze/knowledge-tidy skills → nx_answer + operators (extract/rank/filter/groupby/aggregate via plan_run) + catalog link/resolve/traverse + nx_tidy → memory/scratch → plan-first/plan_match/plan_run.
Swath: store/search/query/nx_answer/all operators/catalog/plan tools + 5-6 skills + the enqueue-hook + aspect worker against the service.

**M3 — Migration/upgrade operator** (`--comprehensive`/`--guided`/`--fullstack`, DONE/in-flight)
Fires: init→serve→index→migrate (cross-model)→parity→rollback + doctor + collection + memory/store/catalog. Already green (comprehensive) + drain in-flight (fullstack).

**M4 — Multi-agent orchestration** (cc-val, real session)
Fires: orchestration skill → dispatch the 10 substantive AGENTS (architect-planner, strategic-planner, developer, test-validator, code-review-expert, substantive-critic, deep-analyst, debugger, codebase-deep-analyzer, deep-research-synthesizer) → SubagentStart hook (conexus subagent-start + sn mcp-inject) → assert each agent returns + its hook injection fired.
Swath: all 10 agents + the subagent hooks in one session.

**M5 — Catalog/provenance + code-nav** (cc-val, real session)
Fires: catalog register/link/links/link_query/resolve/show/update/stats/traverse + serena-code-nav skill + sn grep→serena routing + auto-approve hooks.
Swath: the full catalog tool set + serena routing + permission hooks.

**Union target:** M1∪M2∪M3∪M4∪M5 ≈ the full Tier-1 set and most of the long tail. Remaining singletons after the union (e.g. `daemon_uninstall`, `divergence-language-guard`, SessionEnd launcher) get a tiny dedicated check or an accepted-gap note — explicitly, never silent.

### Per-scenario assertion model
Each scenario asserts the OUTCOMES along the journey (RDR file created, aspect rows drained, agent returned a verdict, routing hook redirected, memory round-tripped) + a coverage ledger (which surfaces fired) so the union is visible, not assumed. Non-vacuity gates (as in `--stress`) prevent a scenario that "ran but touched nothing" from passing.

## Vehicles
- `tests/e2e/migration-rehearsal/run.sh --fullstack` — MCP tools + the real enqueue/worker path against the live 6.0.0 service (M2, M3).
- `tests/cc-validation/` — real `claude` sessions for commands / skills / hooks / agents (M1, M4, M5).
- Building these 5 multivariate scenarios + the union ledger is the 6.0.0 plugin-surface coverage effort (its own RDR/epic).
